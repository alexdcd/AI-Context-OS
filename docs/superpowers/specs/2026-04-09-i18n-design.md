# i18n Design — AI Context OS

**Date:** 2026-04-09  
**Status:** Approved  
**Scope:** UI chrome only (EN/ES). Rust backend strings, YAML frontmatter, folder names, and user-generated content (journal, tasks, notes) are NOT translated.

---

## 1. Library Choice

**i18next + react-i18next**

- Hook: `useTranslation()` → `t("key")`
- Component: `<Trans>` for strings with embedded JSX
- Interpolation syntax: `{{variable}}`
- Fallback language: `"en"` (always)
- TypeScript: `CustomTypeOptions` referencing `en.json` for type-safe keys

---

## 2. Language Preference

Stored **globally** in `settingsStore.ts` (Zustand, persisted to localStorage key `obs-settings`).

```ts
// settingsStore additions
language: "en" | "es"         // default: "en"
setLanguage: (lang) => void   // calls i18n.changeLanguage(lang) internally
```

On app start, `i18n/index.ts` reads `settingsStore.getState().language` and passes it as `lng` to `i18next.init()`. This avoids a render before the language is set (no flicker).

---

## 3. File Structure

```
src/i18n/
  index.ts              ← i18next init (reads from settingsStore, no React dependency)
  locales/
    en.json             ← source of truth
    es.json             ← Spanish translation
  types.d.ts            ← CustomTypeOptions: keys from en.json, auto-complete in all files
```

`main.tsx` imports `src/i18n/index.ts` before `ReactDOM.createRoot()`.

## 5. Component Migration Strategy

### Standard components
```tsx
const { t } = useTranslation()
// Before: <button>New note</button>
// After:  <button>{t("explorer.newNote")}</button>
```

### Object arrays with labels (navItems, TEMPLATES, themeOptions)
Translate the label at render time, not at definition time:
```tsx
const navItems = [
  { key: "explorer", icon: LayoutGrid, path: "/" },
  // no label stored in the array
]
// In JSX: <span>{t(`sidebar.${item.key}`)}</span>
```

### MEMORY_ONTOLOGY_LABELS in types.ts
This is a constant used across components. It must NOT call hooks. Strategy: remove the constant, replace call sites with `t("ontologies.source")` etc. directly. Call sites: grep for `MEMORY_ONTOLOGY_LABELS`.

### Dynamic/interpolated strings
```tsx
t("graph.nodesEdges", { nodes: nodeCount, edges: edgeCount })
t("tasks.overdue", { n: daysOverdue })
t("explorer.deleteFolder", { name: folderName })
```

## 7. settingsStore Changes

```ts
// New field + action
language: "en" | "es"

setLanguage: (lang: "en" | "es") => {
  set({ language: lang })
  i18n.changeLanguage(lang)
}
```

Default: `"en"`. The existing `persist` middleware already handles localStorage.

---

## 8. Onboarding Integration

`OnboardingWizard.tsx` already captures `language: "es" | "en"` in its local state. On workspace creation completion, call `settingsStore.setLanguage(profile.language)`. This syncs the onboarding choice to the global store for all future sessions.

## 9. Language Selector in SettingsView

New "Language" section (between Appearance and Backup):
- Two radio/button options: English / Español
- Calls `settingsStore.setLanguage(lang)` on click
- Language switches immediately (i18next is reactive via react-i18next)

## 10. TypeScript Setup

```ts
// src/i18n/types.d.ts
import enTranslation from './locales/en.json'

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation'
    resources: {
      translation: typeof enTranslation
    }
  }
}
```

This gives autocompletion and compile-time errors for missing/typo'd keys across all 28 files.

---

## 11. Scope Boundaries (NOT translated)

- Rust backend: error messages, system logs → stay in English
- YAML frontmatter fields (`type:`, `tags:`, etc.) → structural, not translated
- Folder names (`inbox/`, `sources/`, `.ai/`) → structural
- User-generated content (journal entries, task notes, memory body) → user's own language
- Date formatting: `date-fns` already supports locale — use `es` locale for Spanish date display (Journal view relative dates: "Ayer", "Hoy")
